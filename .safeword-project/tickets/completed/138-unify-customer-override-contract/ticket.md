# Unify customer override contract in LLM hook (supersedes #019)

## Why this exists

Safeword has two lint contexts per language:

1. **Project config** — `eslint.config.mjs`, `ruff.toml`, `.golangci.yml`. Human-facing: IDE, pre-commit, CI.
2. **LLM hook config** — `.safeword/eslint.config.mjs`, `.safeword/ruff.toml`, `.safeword/.golangci.yml`. Generated by safeword, overwritten on upgrade, used only by the post-tool-lint hook that runs after AI edits.

**The tech debt (current state):** for ESLint the composition is `[...customerConfig, safewordStrictRules, prettier]` — flat-config "later wins" means safeword overrides the customer on any rule safeword sets. A customer who disables `no-unused-vars` in their `eslint.config.mjs` still sees it enforced in the LLM hook. Two surprising consequences:

- Customer's mental model ("my config is my config") doesn't match reality.
- Customer's only remediation today is editing `.safeword/eslint.config.mjs` directly, which gets wiped on every `safeword upgrade`.

For Ruff there's an analogous but narrower gap. The pack has two modes:

- **Mode #2 (pre-existing customer ruff config):** `.safeword/ruff.toml` uses `extend = "../ruff.toml"`. Customer's `ignore` and `select` propagate via ruff's extend mechanism. Working.
- **Mode #1 (no pre-existing ruff config, safeword generates customer's):** `.safeword/ruff.toml` has **no** extend; customer's project `ruff.toml` has `extend = ".safeword/ruff.toml"`. Hook invokes `ruff check --config .safeword/ruff.toml <file>` — which bypasses customer's `ruff.toml` entirely. Customer edits to their own `ruff.toml` do not reach the hook. Silent failure.

## Why 019's approach is insufficient

Ticket [019](../019-customer-override-rules/ticket.md) proposed solving this by adding a **second override surface** — `.safeword-project/eslint-overrides.mjs`, `.safeword-project/ruff-overrides.toml`, `.safeword-project/golangci-overrides.yml` — that safeword would compose AFTER its strict layer, giving customers a "win" slot. That design works mechanically but:

- Creates two surfaces for customers to know about (their project config + safeword's override file). Discoverability gap.
- Inverts precedence contextually: customer's project config wins in the IDE; customer's `.safeword-project/*-overrides.*` wins in the LLM hook. Non-obvious.
- Adds bespoke merge code per linter.

## The unified contract (this ticket)

> The customer's project-owned lint config is authoritative for the LLM hook.

Single surface, one mental model. Every editor, pre-commit, CI, and LLM hook reads the same ruleset from the customer's project config. Safeword's LLM-strict rules still layer in — as _defaults_ the customer can override, not overrides that win.

## Research: why per-linter implementation differs

| Linter           | Native composition       | Path chosen                                 | Rationale                                                                                                                             |
| ---------------- | ------------------------ | ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| ESLint           | Flat config "later wins" | **Flip composition** (Path B)               | Native to ESLint 9.x ([docs](https://eslint.org/docs/latest/use/configure/configuration-files)). One-line template change. Idiomatic. |
| Ruff             | `extend` directive       | **Unify extend-direction** (Path C variant) | `.safeword/ruff.toml` always extends customer. Leverages ruff's native extend. Resolves Mode #1 gap.                                  |
| golangci-lint v2 | No composition model     | Path C (status quo)                         | Already uses unionArrays + fillGapMerge ([additive-config memory](../../../CLAUDE.md)). No change needed.                             |
| Clippy           | No composition model     | Path C (status quo)                         | Fill-gap merge via CLIPPY_CONF_DIR. No change needed.                                                                                 |
| Sqlfluff         | Key-by-key patching      | Path C (status quo)                         | Omits customer-owned keys. No change needed.                                                                                          |

**Divergence is unavoidable** (each ecosystem's native primitives differ), but the **customer-facing contract is unified**: "edit your project config, your rules win."

## Implementation plan

### 1. ESLint — flip composition

**File:** `packages/cli/src/templates/config.ts`

Change `getSafewordEslintConfigExtending()` template string:

```diff
 export default [
-  ...projectConfig,
-  safewordStrictRules,
+  safewordStrictRules,
+  ...projectConfig,
   ${prettier.configEntry}
 ];
```

Same flip for `getSafewordEslintConfigLegacy()` and `getSafewordEslintConfigStandalone()`. In the standalone case the "project config" slot is the generated root `eslint.config.mjs`; same flip applies to the generated-root template for consistency.

**Note on `eslint-config-prettier`:** keep it last. It disables stylistic rules that conflict with Prettier; we want those off in the LLM hook regardless of customer config.

### 2. Ruff — unify extend direction

**File:** `packages/cli/src/packs/python/files.ts`

Change `generateRuffBaseConfig(undefined)` (the standalone branch) to generate `.safeword/ruff.toml` with `extend = "../ruff.toml"` and the project-root `ruff.toml` to be the customer's pristine owned file (no extend, customer adds whatever they like).

After change, both modes produce the same `.safeword/ruff.toml` shape — which always extends customer's project `ruff.toml`, which is always customer-owned.
