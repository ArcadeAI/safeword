# Feature: Gherkin for TypeScript Projects

**Type:** Feature | **Priority:** Backlog | **Parent:** Epic 102

## Problem

SafeWord orchestrates BDD (phases 0-7) but never executes the specs it produces. Phase 4 writes Given/When/Then scenarios in Markdown (test-definitions.md), which are manually translated into Vitest tests in Phase 6. The scenarios and tests are disconnected — Markdown drifts from reality.

This affects both SafeWord's own development and any TypeScript customer project using SafeWord's BDD workflow.

## Solution

Add QuickPickle (Vitest Gherkin plugin) so `.feature` files run as native Vitest tests. SafeWord uses this internally first, then scaffolds the same setup into customer TypeScript projects.

## Architecture Decisions

### Runner: QuickPickle

- True Vitest plugin — `.feature` files become tests via the Vite plugin pipeline
- ESM-native, official `@cucumber/gherkin` parser, 11 deps / 297 KB
- Rejected: `@cucumber/cucumber` (40 deps, own runner), `@amiceli/vitest-cucumber` (CJS, manual wiring), `@cucumber/node` (pre-1.0, `node:test` only)
- Zero new runners — `bun test` still runs everything
- **Bus-factor risk:** ~50 stars, solo maintainer (2026-04). Re-evaluate at implementation time. Architecture is runner-agnostic

### Layer: Complement, not replace

- Gherkin at the **acceptance layer** — user-facing behavior in business language
- Unit and integration tests stay in Vitest `.test.ts` unchanged
- Double-loop: outer = failing Gherkin scenario; inner = TDD with Vitest
- New features only — do NOT retrofit existing tests (anti-pattern: feature files after code)

### Artifact: .feature replaces scenarios in test-definitions.md

- `.feature` files are the single source of truth for Given/When/Then
- `test-definitions.md` becomes planning-only: scope, open questions, rationale, progress checkboxes referencing scenario names
- No auto-generation between formats — that enshrines duplication
- No behavioral scenarios in both places (documentation drift anti-pattern)

### Step definitions: Domain-grouped with barrel file

- QuickPickle loads steps via Vitest `setupFiles` — one barrel imports domain files
- Organize by domain concept (config, hooks, cli), NOT by feature file
- Steps are global — any feature can use any step (ubiquitous language)

### Shared steps: Thin core, grow organically

- Start with ~5-7 infrastructure steps the first feature needs
- Do NOT pre-build parameterized variants until duplication emerges
- Step definitions stay thin — extract logic into helper functions
- "Duplication is far cheaper than the wrong abstraction" (Sandi Metz)

## SafeWord Internal File Structure

```
packages/cli/
  features/                    <- .feature files by domain
    config/
      config-init.feature
    hooks/
      hook-lifecycle.feature
    cli/
      cli-check.feature
  steps/                       <- step definitions by domain
    index.ts                   <- barrel file (the setupFile)
    world.ts                   <- custom QuickPickleWorld subclass
    shared.steps.ts            <- filesystem, temp dirs, assertions
    config.steps.ts            <- config domain steps
    hooks.steps.ts             <- hooks domain steps
    cli.steps.ts               <- CLI execution steps
```

## Customer TS Project Scaffold (generated by `safeword setup`)

```
customer-ts-project/
  .safeword.yml                  <- bdd.enabled: true
  features/                      <- .feature files
    login.feature
  steps/                         <- TypeScript step definitions
    index.ts                     <- barrel file
    world.ts                     <- shared test state
    shared.steps.ts              <- run commands, assert output, file I/O
  package.json                   <- QuickPickle + Vitest as devDeps
  vitest.config.ts               <- QuickPickle plugin configured
```

## Initial Shared Step Vocabulary

```gherkin
Given a temporary project directory
Given a file {string} with content:
When I run {string}
Then the exit code should be {int}
Then stdout should contain {string}
Then stderr should contain {string}
# After hook for temp directory cleanup
```

## Implementation Steps

### 1. Install and configure QuickPickle

- `bun add -d quickpickle`
- Add QuickPickle plugin to `vitest.base.ts` (shared base — all configs inherit)
- Add `setupFiles: ['./steps/index.ts']` to `vitest.base.ts`
- Add `.feature` inclusion to `vitest.config.ts` only (not slow/release configs)
- Verify `.feature` files are discovered and run (empty feature = passing)

### 2. Create step definition scaffold

- `steps/index.ts` barrel file
- `steps/world.ts` with custom `QuickPickleWorld` subclass (temp dir, CLI result state)
- `steps/shared.steps.ts` with the 5-7 infrastructure steps
- Wire existing test helpers (`runCli`, `createTemporaryDirectory`) into step definitions

### 3. Write first .feature file (proof of concept)

- Pick one existing behavior (e.g., `safeword check` health check) and write as `.feature`
- One-time validation that the setup works — not a pattern for all existing tests
- Existing Vitest test stays (different abstraction level)
- Verify it runs via `bun test`

### 4. Update BDD skill Phase 4 and Phase 6

- Phase 4: produce `.feature` files as the behavioral artifact (not Markdown Given/When/Then)
- Phase 6: write step definitions for `.feature` scenarios, then TDD the implementation
- Update `test-definitions.md` template to reference `.feature` files instead of containing scenarios
- Update `.claude/skills/bdd/TDD.md` with the new workflow

### 5. Add second feature to validate architecture

- Pick a different domain (e.g., hooks or config)
- Write `.feature` + domain-specific steps
- Validate shared steps reuse cleanly
- Extract new shared steps only if duplication emerges naturally

### 6. Customer TS project scaffolding

- Add `bdd.enabled: true` flag to `.safeword.yml` (opt-in)
- `safeword setup` scaffolds: `features/`, `steps/`, QuickPickle devDep, vitest config
- `safeword check` verifies Gherkin infrastructure is wired up
- Hook integration: pre-commit/pre-push can run `.feature` specs via `bunx vitest --run features/`

## Out of Scope

- Non-TypeScript projects — see ticket 102b
- Native-language step definitions — see ticket 102c
- Migrating existing Vitest tests to Gherkin
- Browser/UI testing (no Playwright BDD)
- Auto-generation between Markdown and .feature files
- Gherkin for unit-level tests
