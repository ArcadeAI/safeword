# Improve learnings discoverability for Claude Code agents

**Goal:** Make the `.safeword-project/learnings/` folder self-discoverable via a Karpathy-style LLM Wiki pattern (plain markdown + auto-maintained INDEX.md + SAFEWORD.md instruction), with no hard scaling ceiling and portable across any agent that reads CLAUDE.md / AGENTS.md-style configs.

**Why:** Hard-won project lessons sit in `.safeword-project/learnings/` but weren't being used. The original "check FIRST when stuck" conditional-recall line in SAFEWORD.md degraded after long sessions. Even when an agent did look, cryptic filenames gave no signal for which file was relevant.

## Design

### Mechanism — Karpathy-aligned with one concession

Follow [Karpathy's LLM Wiki pattern](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f): plain markdown files, one per concept, navigable via an LLM-maintained `INDEX.md` + grep. No vector DB, no platform-specific skill wrapping.

**The concession to our infrastructure:** we auto-generate `INDEX.md` deterministically from `Covers:` lines instead of having the LLM maintain it conversationally. Strictly better for a team setting — no drift, reviewable diffs, fast.

### Files

```
.safeword-project/learnings/
  INDEX.md                    ← auto-generated catalog, no size cap
  <learning>.md               ← each has `Covers:` on line 3
```

### The generated INDEX.md shape

```markdown
# Project Learnings — Index

<!-- Auto-generated by `safeword sync-learnings`. Do not edit this file by hand. -->

## Learnings (N)

- **<Title>** — <Covers: topics>
  → `.safeword-project/learnings/<file>.md`
```

Agent reads this whole file via the Read tool (no field-size cap applies).

### Auto-generation

A single CLI command — `safeword sync-learnings` — reads each `.md` in the folder, extracts the `Covers:` line (line 3), regenerates `INDEX.md`. Deterministic string extraction. No AI, no fuzzy matching.

**One hook fires it:** `PostToolUse` on Edit/Write/MultiEdit/NotebookEdit matching `.safeword-project/learnings/*.md`. That's enough:

- In-session edits: PostToolUse catches them
- Out-of-band edits (git pull, VS Code): next Claude Code session reloads `CLAUDE.md`/`SAFEWORD.md` fresh, which instruct the agent to read `INDEX.md` before work. If index lags real content briefly, the agent still reads the underlying files via grep. Low-impact drift.

### SAFEWORD.md instruction

Five-line block teaches the agent two things: (1) read `INDEX.md` before non-trivial work, (2) add new learnings when solving something non-obvious.

**Compaction survival (dogfood + customer parity):**

Split `AGENTS_MD_LINK` into two constants so each file gets the right kind of reference:

- **`CLAUDE_MD_IMPORT_BLOCK`** — prepended to CLAUDE.md. Uses `@./.safeword/SAFEWORD.md` (real import). On launch and on every `/compact`, [Claude Code re-reads project-root CLAUDE.md](https://code.claude.com/docs/en/memory) and re-expands `@` imports — SAFEWORD.md content inlines fresh into the compaction-resistant slice of context.
- **`AGENTS_MD_LINK`** — prepended to AGENTS.md (unchanged prose). AGENTS.md is for Cursor/Gemini/other agents reading the file directly; they don't use Claude Code's import syntax.

**Why we didn't use `@` import in both files:** Claude Code does not deduplicate imports ([GitHub issue #21891](https://github.com/anthropics/claude-code/issues/21891)). If CLAUDE.md `@`-imports AGENTS.md and AGENTS.md has `@./.safeword/SAFEWORD.md`, SAFEWORD.md appears twice. Only CLAUDE.md gets the `@` import.

**Why this is load-bearing:** without the `@` import, the customer mechanism is a 4-step cross-file delegation chain (CLAUDE.md → read SAFEWORD.md → read INDEX.md → read specific file). Our own learning (`.safeword-project/learnings/instruction-attention-hierarchy.md`) measures cross-file delegation at ~20% compliance. The `@` import collapses the first two hops into a zero-decision inline expansion, bringing the chain back to parity with v1's 2-step auto-invocation.

Future refactors: don't stack more indirection layers blindly; the 5-hop import cap applies. And if `@./AGENTS.md` is ever added to CLAUDE.md (possible path for Cursor parity), drop the AGENTS.md prose to avoid double-loading SAFEWORD.md via two paths.

### Covers: convention

Every learning file MUST have `Covers: <topic list>` on line 3.

- `safeword sync-learnings` is **lenient**: skips non-conforming files with a stderr warning
- `safeword audit` flags them as W006

### Why not the Agent Skill path (discarded)

An earlier iteration (committed 2026-04-17) wrapped the folder in a `project-learnings` skill with an auto-generated `description:` field. Discarded because:

1. **1024-char description cap caused a scaling cliff** — truncation silent past ~15 learnings
2. **Compaction hierarchy** — CLAUDE.md re-reads fresh after `/compact`; skills fit in a shared 25k-token re-attachment budget and can be dropped
3. **Platform lock-in** — skill was Claude-Code-specific; INDEX.md works across any markdown-aware agent (Cursor, Aider, Gemini CLI, etc.)
4. **Karpathy-pure is simpler** — ~200 fewer lines of code, no budget math, no truncation logic, no per-file length checks

### Not doing (with justifications)

- **No per-learning skills** — folder-of-markdown + INDEX.md is simpler and scales further
- **No `.claude/skills/project-learnings/` wrapper** — tried it first, removed after Karpathy-pattern research showed the skill adds complexity for a ceiling we don't want
- **No keyword-based auto-invocation** — would require putting topics in a skill description (1024-char cap); replaced by CLAUDE.md/SAFEWORD.md instruction, which research shows survives compaction and has high adherence when focused
- **No content reorg** (splits, renames, condensation) — learnings load on demand via Read tool; size doesn't matter
- **No `head -3` convention in SAFEWORD.md** — still conditional recall; deleted entirely in favor of "read INDEX.md"
- **No subdirectory organization (`eslint/`, `testing/`, ...)** — out of scope; customers can add it by convention later
- **No `log.md` (Karpathy's append-only timeline)** — nice-to-have, not required. Future ticket if wanted.
