# Enrich BDD phase-transition handoff artifacts

**Goal:** When the BDD skill transitions between major phases (especially scenario-gate → implement), write a structured handoff brief so context survives compaction without degradation.

**Why:** Anthropic's April 2026 harness research found "complete context resets with structured handoffs preserve agent clarity" better than compaction. Currently, the compaction hook re-injects a minimal one-liner (`Ticket: {id} | Phase: {phase} | Gate: {gate}`). For long BDD sessions (20+ scenarios), discovery/scenario context may degrade after compaction, leaving the implementing agent without understanding of _why_ scenarios were written that way.

## Context

### Current state

- `session-compact-context.ts` re-injects a one-liner after compaction
- Phase state is tracked in ticket.md frontmatter (`phase:` field)
- Artifact gates prevent phase skipping (pre-tool-quality.ts)
- Dogfooding data: 30+ commit sessions complete successfully with current model

### Research backing

- Anthropic "Harness Design: Long-running apps" (April 2026): Planner→Generator→Evaluator with context resets
- Anthropic "Effective Context Engineering" (2025): context rot degrades accuracy as transcript grows
- Agent-behavior-research.md: "Context resets with structured handoffs preserve agent clarity"

### Proposed approach (from brainstorm)

At major phase boundaries (intake→define-behavior, scenario-gate→implement), write a structured handoff section to the ticket's work log containing:

- Validated scenarios with rationale (why each matters)
- Architectural decisions made during discovery
- Edge cases discussed and explicitly excluded
- Test scope decisions

This gives the implement phase everything it needs even if compaction wipes the discovery conversation. The handoff is written to the ticket (persistent artifact), not injected into context (transient).

### Alternatives considered

- **`context: fork` for implement phase**: Clean context but loses conversational nuance about _why_ scenarios exist. More architectural change than needed.
- **Phase-boundary conversation resets**: Most aligned with Anthropic research but adds friction (user must start new conversations). Could be automated but changes UX.
- **Do nothing**: Current model works for 30-commit sessions. Unknown behavior beyond that. This ticket is the incremental improvement.

### Open questions

- What's the right format for the handoff brief? Structured YAML in frontmatter vs. markdown section in work log?
- Should the handoff be auto-generated by a hook at phase transition, or part of the skill's phase-exit instructions?
- Is there a maximum useful size for handoff briefs before they become their own context burden?
