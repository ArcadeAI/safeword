# Dimensions: plan-implementation phase before TDD

Derived from done_when, the seven intake decisions, and the #480 investigation (phase-machinery + gate phase-lists).

| Dimension                            | Partitions                                                                                                                                                                                                                                                         |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Phase-transition legality            | scenario-gate → plan-implementation (one step, legal); scenario-gate → implement (now a skip — denied without `phase_skips`); plan-implementation → implement (legal when gated conditions met); backward moves (always legal); intake → done jump (skip list names 5 phases) |
| Transition-gate plan validation      | impl-plan.md missing; present but invalid (missing section / bad status line); valid with status `planned`; valid but status `implemented` (stale from a replan loop — boundary)                                                                                     |
| Ticket-kind applicability            | new-flow feature (spec.md present) — gated; task — exempt; legacy feature without spec.md — grandfathered (matches M6D315 exemptions)                                                                                                                                |
| Stop-gate cumulative artifacts       | at plan-implementation: test-definitions.md required (decision 3), impl-plan.md NOT required (decision 6 — work-in-progress stops legal); at implement/verify/done: impl-plan requirements unchanged (M6D315)                                                        |
| Code-edit blocking during planning   | application code at plan-implementation (denied — decision 2); ticket/meta artifacts incl. impl-plan.md (allowed); application code at implement with test-defs (allowed, unchanged)                                                                                 |
| Guidance routing                     | per-phase prompt reminder (planning line, the #530 anchor); resume-table row; PHASE_EVIDENCE quality message; phase-file table → PLAN_IMPLEMENTATION.md                                                                                                              |
| Denial-message quality               | names the missing artifact + concrete next action, plain language (NTB-readable, no internal jargon)                                                                                                                                                                 |
| Doc-content separation               | scenario-gate exit contains no implementation-design steps; PLAN_IMPLEMENTATION.md owns impl-plan authoring; zero shipped "authored at scenario-gate exit" strings                                                                                                   |
| Enum / surface completeness          | CANONICAL_PHASES order; BddPhase + PHASE_EVIDENCE (compile-exhaustive); prompt-reminders map (silent-miss risk); LEDGER_REQUIRED_PHASES; SPLITTING.md checkpoint row + child restart target (decision 7)                                                             |
| Cross-harness parity                 | schema ownedFiles entry (Claude); CODEX_SKILL_TEMPLATE_FILES row; CURSOR_RULE_WRAPPERS entry; byte-identical template ↔ dogfood trio + Cursor rule pair                                                                                                              |
| ADR record                           | superseding ADR present in ARCHITECTURE.md; decomposition-retirement ADR marked superseded (not silently contradicted)                                                                                                                                               |

Boundary values: the two adjacent transitions around the inserted phase (scenario-gate → plan-implementation, plan-implementation → implement) and the `planned` vs `implemented` status line on an otherwise-valid plan.

Partitions covered elsewhere (no scenario by design): backward moves — already proven by `features/phase-provenance.feature` ("Rework … never blocks"); PHASE_EVIDENCE quality message — compile-proven (`Record<BddPhase, string>` is exhaustive; a missing entry fails `tsc`, no runtime scenario can add signal).
