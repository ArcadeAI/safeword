## Slash Commands

Quick actions you type in Claude Code or Cursor. Just type the command and hit enter. Codex uses plugin-scoped skills such as `safeword:bdd`, `safeword:verify`, and `safeword:explain`.

| Command            | What it does                              |
| ------------------ | ----------------------------------------- |
| `/lint`            | Fix code style issues                     |
| `/audit`           | Find dead code and architecture drift     |
| `/quality-review`  | Deep review against latest research       |
| `/cleanup-zombies` | Kill stuck processes hogging ports        |
| `/bdd`             | Start BDD workflow (includes TDD)         |
| `/debug`           | Start structured debugging workflow       |
| `/refactor`        | Start safe refactoring workflow           |
| `/verify`          | Run completion checklist for current task |

## Hooks

Hooks run automatically at key moments. You don't invoke them—they work in the background.

<Tabs syncKey="editor">
  <TabItem label="Claude Code">
    | When                    | What happens                                        |
    | ----------------------- | --------------------------------------------------- |
    | Session starts          | Verifies AGENTS.md link, shows version, checks lint |
    | You submit a prompt     | Adds timestamp, reminds agent to ask questions      |
    | Agent edits a file      | Runs the appropriate linter, enforces quality gates  |
    | Agent edits Go/Python/TS/Rust | Points it at that language's coding-skill (first edit per scenario) |
    | You stop the session    | Prompts for quality review                          |

    ### Customizing hooks

    Default hooks get overwritten on `safeword upgrade`. To customize:

    1. Copy the hook to your project namespace root (`.project/` by default — see [Configuration](/reference/configuration#project-namespace-root))
    2. Update `.claude/settings.json` to point to your copy
    3. Your custom hook stays safe from upgrades

    Hook config: `.claude/settings.json`
    Hook files: `.safeword/hooks/`

  </TabItem>
  <TabItem label="Cursor">
    Safeword installs hooks (`.cursor/hooks.json`) and rules (`.cursor/rules/`) to guide your agent's behavior.

    | When                    | What happens                    |
    | ----------------------- | ------------------------------- |
    | Agent edits a file      | Runs the appropriate linter     |
    | Agent edits Go/Python/TS/Rust | Coding-skill nudge (dormant — Cursor bug #534) |
    | You stop the session    | Prompts for quality review      |

    ### Rules

    Rules shape agent behavior through system prompts:

    | Rule              | What it does                                             |
    | ----------------- | -------------------------------------------------------- |
    | Core              | Code philosophy, anti-patterns, quality standards        |
    | Debugging         | Investigate before jumping to fixes                      |
    | Refactoring       | One small change at a time, test after each              |
    | Quality Reviewing | Check against latest research and documentation          |
    | Testing           | Behavioral testing principles and test type selection    |
    | Ticket System     | Context anchoring with tickets and work logs             |
    | BDD (8 rules)     | Full feature workflow from discovery through TDD to done |

    ### Customizing hooks and rules

    Default hooks and rules get overwritten on `safeword upgrade`. To customize:

    1. Copy the file to your project namespace root (`.project/` by default)
    2. For rules: reference your copy in your workflow
    3. For hooks: update `.cursor/hooks.json` to point to your copy

    Hook config: `.cursor/hooks.json`
    Rule files: `.cursor/rules/`

  </TabItem>
  <TabItem label="Codex">
    Safeword's Codex support is package/plugin-backed. Project setup writes hook config (`.codex/config.toml`) that calls packaged Safe Word commands; the Codex plugin provides scoped workflow skills.

| When                            | What happens                                              |
| ------------------------------- | --------------------------------------------------------- |
| Session starts                  | Loads packaged Safe Word standing instructions            |
| Before supported tools          | Runs the packaged `safeword hook codex pre-tool-use` gate |
| Agent finishes a supported tool | Surfaces queued Safe Word post-tool context               |
| You submit a prompt             | Surfaces queued Safe Word prompt context                  |
| You stop the session            | Emits queued Stop continuation nudges                     |

    Safeword's Codex edit-gate command covers the documented PreToolUse paths it configures: `Bash`, `apply_patch` edit payloads, and file-editing tools. Live Codex runs can also report `file_change` execution items; Safeword records that as a Codex runtime boundary, not as a PreToolUse path it claims to guard. The Codex Stop command is separate: it uses continuation output for done-phase reminders such as ARCHITECTURE.md drift.

    ### Skills

    Codex reads safeword workflows from the Safe Word Codex plugin. The canonical invocation names are scoped, for example `safeword:bdd`, `safeword:verify`, and `safeword:explain`. Safeword no longer installs Safe Word-owned repo-local aliases under `.agents/skills/`.

    ### Hook trust

    Codex hooks only run after you review and trust them in Codex. Safeword writes the project config and ships plugin hook metadata, but Codex owns the trust decision.

    Hook config: `.codex/config.toml`

Hook commands: `npx --yes safeword hook codex <event>`

    Plugin skill files: bundled in the Safe Word package under `codex-plugin/skills/`

### Auto-linting

After your agent writes or modifies a file, the right linter runs:

| File type               | Linter        |
| ----------------------- | ------------- |
| TypeScript / JavaScript | ESLint        |
| Python                  | Ruff          |
| Go                      | golangci-lint |
| Rust                    | clippy        |
| SQL / dbt               | SQLFluff      |

If a linter isn't installed, it's silently skipped.

**What you see:** Usually nothing — the linter auto-fixes quietly. If problems remain, your agent sees them and fixes them immediately.

## Language coding-skills

When safeword sets up (or upgrades) a project, it installs a small third-party **coding-skill** for each language it detects, and nudges your agent toward that skill the first time you edit the language in a scenario.

| Language   | Skill source (MIT)         |
| ---------- | -------------------------- |
| Go         | `jeffallan/golang-pro`     |
| Python     | `jeffallan/python-pro`     |
| TypeScript | `jeffallan/typescript-pro` |
| Rust       | `leonardomso/rust-skills`  |

Language coding-skills install into `.claude/skills/` and, where supported by the agent, `.agents/skills/` via the `npx skills` CLI. These are third-party language helpers, not Safe Word Codex workflow aliases. The install is **best-effort** — a missing network or installer error degrades to a warning and never blocks setup. It runs once; later upgrades skip it when the skill is already present.

**What you see:** edit a `.go` / `.py` / `.ts` / `.tsx` / `.rs` file and, in Claude Code, a one-line reminder points your agent at the matching skill and surfaces the skill's own description. It fires once per scenario and stays silent for other file types. The Cursor adapter is built but dormant until Cursor fixes an upstream injection bug ([#534](https://github.com/TheMostlyGreat/safeword/issues/534)). Codex's Safe Word workflow guidance now comes from the Codex plugin rather than repo-local `.agents/skills/` aliases.

To skip the skill install, set `SAFEWORD_SKIP_SKILLS=1` (or `SAFEWORD_SKIP_INSTALL=1` to skip all network installs during setup).

Frontier models already write most core-language idioms unaided, so treat this as a light nudge, not a transformation.

## Skills

Skills are structured workflows that activate based on what you're asking for. Claude Code uses `.claude/skills/`; Codex uses the Safe Word Codex plugin's scoped skills. When you say "debug this test failure" or "refactor this function", your agent follows the relevant methodology instead of jumping straight to code changes.

| Skill             | Triggers on                                                                                                         |
| ----------------- | ------------------------------------------------------------------------------------------------------------------- |
| BDD Orchestration | "add", "implement", "build", "feature", "iteration", "story", "phase", "resume", "continue" (includes TDD workflow) |
| Debugging         | "debug", "fix this error", "test is failing"                                                                        |
| Quality Reviewing | "review", "check security", "verify versions"                                                                       |
| Refactoring       | "refactor", "clean up", "extract", "rename"                                                                         |
| Testing           | "write tests", "add tests", "test this", "improve coverage" (also invoked by other skills internally)               |
| Ticket System     | "create ticket", "work log", "resume", ticket ID references (context anchoring for complex work)                    |

### BDD Orchestration

Full feature workflow: **Discovery** → **Scenarios** → **Scenario Gate** → **Plan Implementation** → **TDD** → **Verify** → **Done**. Includes TDD (RED/GREEN/REFACTOR) inline. Large features can be split at any checkpoint.

**Example:** "Add user authentication to the API"

Your agent will discover requirements, write BDD scenarios, validate them against quality criteria (atomic, observable, deterministic), plan the implementation (impl-plan.md — proof plan, build order, architecture alignment), then implement each scenario with test-driven development.

### Debugging

Investigates before fixing. Follows: **Investigate** → **Hypothesize** → **Test** → **Fix**.

**Example:** "This test is failing but I don't know why"

Instead of immediately changing code, your agent will read the test, trace the execution path, form a hypothesis, and verify it before making changes.

### Quality Reviewing

Deep review that checks your code against latest research using web search.

**Example:** "Review this authentication code"

Your agent will check against official library docs, look for known security issues, and verify you're using current versions.

### Refactoring

Small, safe changes with test verification after each step: **One change** → **Test** → **Commit**.

**Example:** "Clean up this messy function"

Your agent will make one small improvement, run tests, commit, then repeat—never batching multiple changes.

### Testing

Core knowledge for writing good tests. Ensures tests verify observable behavior, not implementation details.

**Example:** "Write tests for the payment module"

Your agent will choose the right test type (unit, integration, E2E), write focused assertions, and avoid brittle mocking. This skill is also invoked automatically by other skills — BDD at the RED phase, refactoring for characterization tests, and debugging.

### Ticket System

Context anchoring for complex, multi-step work. Creates tickets with colocated artifacts (specs, test definitions, work logs) in `.project/tickets/` (legacy installs: `.safeword-project/tickets/`).

**Example:** "Create a ticket for the auth refactor"

Your agent will create a structured ticket folder with a spec, test definitions, and work log — preventing context loss across long tasks.
