## Slash Commands

Quick actions you type in Claude Code or Cursor. Just type the command and hit enter.

| Command            | What it does                              |
| ------------------ | ----------------------------------------- |
| `/lint`            | Fix code style issues                     |
| `/audit`           | Find dead code and architecture drift     |
| `/quality-review`  | Deep review against latest best practices |
| `/cleanup-zombies` | Kill stuck processes hogging ports        |
| `/bdd`             | Start BDD workflow (includes TDD)         |
| `/refactor`        | Start safe refactoring workflow           |
| `/done`            | Run completion checklist for current task |

## Hooks

Hooks run automatically at key moments. You don't invoke them—they work in the background.

<Tabs syncKey="editor">
  <TabItem label="Claude Code">
    | When                    | What happens                                        |
    | ----------------------- | --------------------------------------------------- |
    | Session starts          | Verifies AGENTS.md link, shows version, checks lint |
    | You submit a prompt     | Adds timestamp, reminds agent to ask questions      |
    | Agent edits a file      | Runs the appropriate linter                         |
    | You stop the session    | Prompts for quality review                          |

    ### Auto-linting

    After your agent writes or modifies a file, the right linter runs:

    | File type               | Linter        |
    | ----------------------- | ------------- |
    | TypeScript / JavaScript | ESLint        |
    | Python                  | Ruff          |
    | Go                      | golangci-lint |
    | Rust                    | clippy        |

    If a linter isn't installed, it's silently skipped.

    **What you see:** `SAFEWORD: Linting...` appears in output. If there are issues, your agent sees them and fixes immediately.

    ### Customizing hooks

    Default hooks get overwritten on `safeword upgrade`. To customize:

    1. Copy the hook to `.safeword-project/` (e.g., `my-lint.ts`)
    2. Update `.claude/settings.json` to point to your copy
    3. Your custom hook stays safe from upgrades

    Hook config: `.claude/settings.json`
    Hook files: `.safeword/hooks/`

  </TabItem>

    | When                    | What happens                    |
    | ----------------------- | ------------------------------- |
    | Agent edits a file      | Runs the appropriate linter     |
    | You stop the session    | Prompts for quality review      |

    ### Auto-linting

    After your agent writes or modifies a file, the right linter runs:

    | File type               | Linter        |
    | ----------------------- | ------------- |
    | TypeScript / JavaScript | ESLint        |
    | Python                  | Ruff          |
    | Go                      | golangci-lint |
    | Rust                    | clippy        |

    If a linter isn't installed, it's silently skipped.

    **What you see:** `SAFEWORD: Linting...` appears in output. If there are issues, your agent sees them and fixes immediately.

    ### Rules

    Rules shape agent behavior through system prompts:

    | Rule              | What it does                                             |
    | ----------------- | -------------------------------------------------------- |
    | Core              | Code philosophy, anti-patterns, quality standards        |
    | Debugging         | Investigate before jumping to fixes                      |
    | Refactoring       | One small change at a time, test after each              |
    | Quality Reviewing | Check against current docs and best practices            |
    | BDD (7 rules)     | Full feature workflow from discovery through TDD to done |

    ### Customizing hooks and rules

    Default hooks and rules get overwritten on `safeword upgrade`. To customize:

    1. Copy the file to `.safeword-project/`
    2. For rules: reference your copy in your workflow
    3. For hooks: update `.cursor/hooks.json` to point to your copy

    Hook config: `.cursor/hooks.json`
    Rule files: `.cursor/rules/`

## Skills

Skills are structured workflows that activate based on what you're asking for. When you say "debug this test failure" or "refactor this function", your agent follows the skill's methodology instead of jumping straight to code changes.

| Skill             | Triggers on                                                                                                         |
| ----------------- | ------------------------------------------------------------------------------------------------------------------- |
| BDD Orchestration | "add", "implement", "build", "feature", "iteration", "story", "phase", "resume", "continue" (includes TDD workflow) |
| Debugging         | "debug", "fix this error", "test is failing"                                                                        |
| Quality Reviewing | "review", "check security", "verify versions"                                                                       |
| Refactoring       | "refactor", "clean up", "extract", "rename"                                                                         |

### BDD Orchestration

Full feature workflow: **Discovery** → **Scenarios** → **Decomposition** → **TDD** → **Splitting** → **Done**. Includes TDD (RED/GREEN/REFACTOR) inline.

**Example:** "Add user authentication to the API"

Your agent will discover requirements, write BDD scenarios, decompose into tasks, then implement each with test-driven development.

### Debugging

Investigates before fixing. Follows: **Investigate** → **Hypothesize** → **Test** → **Fix**.

**Example:** "This test is failing but I don't know why"

Instead of immediately changing code, your agent will read the test, trace the execution path, form a hypothesis, and verify it before making changes.

### Quality Reviewing

Deep review that checks your code against current best practices using web search.

**Example:** "Review this authentication code"

Your agent will check against official library docs, look for known security issues, and verify you're using current versions.

### Refactoring

Small, safe changes with test verification after each step: **One change** → **Test** → **Commit**.

**Example:** "Clean up this messy function"

Your agent will make one small improvement, run tests, commit, then repeat—never batching multiple changes.
