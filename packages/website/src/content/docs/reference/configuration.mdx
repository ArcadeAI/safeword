Most projects work great with the defaults. This page covers customization when you need it.

## ESLint Setup

Safeword includes an ESLint plugin with sensible defaults. Your `eslint.config.mjs`:

```js
import { defineConfig } from 'eslint/config';
import safeword from 'safeword/eslint';

export default defineConfig([
  ...safeword.configs.recommendedTypeScript,
  ...safeword.configs.vitest, // if using Vitest
]);
```

Safeword's generated config uses [`defineConfig`](https://eslint.org/docs/latest/use/configure/configuration-files#the-defineconfig-helper) (ESLint ≥10.4) — the canonical flat-config helper. It enables file-scoped `extends:` and validates the config shape at lint time.

**Available configs:**

| Config                       | Use case                                                                                             |
| ---------------------------- | ---------------------------------------------------------------------------------------------------- |
| `recommended`                | JavaScript only                                                                                      |
| `recommendedTypeScript`      | TypeScript                                                                                           |
| `recommendedTypeScriptReact` | TypeScript + React                                                                                   |
| `recommendedTypeScriptNext`  | TypeScript + Next.js                                                                                 |
| `nextOnlyRules`              | Next.js rules for monorepo file scoping                                                              |
| `vitest`                     | Add Vitest support                                                                                   |
| `playwright`                 | Add Playwright support                                                                               |
| `astro`                      | Add Astro support                                                                                    |
| `tailwind`                   | Add Tailwind support                                                                                 |
| `tanstackQuery`              | Add TanStack Query                                                                                   |
| `storybook`                  | Add Storybook support                                                                                |
| `turbo`                      | Add Turborepo support                                                                                |
| `cli`                        | CLI tools, build tools, scripts — disables security rules that flag dynamic paths and shell commands |
| `relaxedTypes`               | JSON/YAML parsing, API clients, config files — disables strict TypeScript rules for untyped data     |

Apply the `cli` and `relaxedTypes` presets globally by adding them to the array:

```js
export default defineConfig([
  ...safeword.configs.recommendedTypeScript,
  safeword.configs.cli, // if building a CLI tool
  safeword.configs.relaxedTypes, // if handling untyped external data
]);
```

Or scope them to a subset of files using `extends:` — useful when only scripts or config loaders need the relaxations:

```js
export default defineConfig([
  ...safeword.configs.recommendedTypeScript,
  {
    files: ['scripts/**', 'config/**'],
    extends: [safeword.configs.cli, safeword.configs.relaxedTypes],
  },
]);
```

To add your own rules or override individual rules directly, add another entry to the array:

```js
export default defineConfig([
  ...safeword.configs.recommendedTypeScript,
  {
    rules: {
      'no-console': 'warn',
      'security/detect-non-literal-fs-filename': 'off', // override a single rule
    },
  },
]);
```

## Language Detection

Safeword auto-detects your stack and configures linting accordingly:

| Language       | Linting           | Type Checking | Architecture       |
| -------------- | ----------------- | ------------- | ------------------ |
| TypeScript     | ESLint + Prettier | tsc           | dependency-cruiser |
| Python (beta)  | Ruff              | mypy          | import-linter      |
| Go (beta)      | golangci-lint     | go vet        | —                  |
| Rust (beta)    | clippy            | rustc         | —                  |
| SQL/dbt (beta) | SQLFluff          | —             | —                  |

Detection is stored in `.safeword/config.json`. To force a refresh, delete that file and run `safeword setup` again.

## Biome & Alternative Formatters

If your project uses Biome, dprint, or Rome, Safeword detects it automatically and skips Prettier installation. Your formatter keeps running as-is.

**Safeword still installs ESLint.** This is intentional — Biome doesn't support:

- **Security scanning** — catches unsafe patterns like `eval()`, non-literal regex, prototype pollution
- **Cyclomatic complexity** — flags overly complex functions that AI agents tend to generate
- **Framework-specific rules** — React hooks, Next.js best practices, Astro components

Both tools coexist cleanly — ESLint handles code analysis (bugs, security, complexity) while Biome handles formatting. Safeword configures Biome to ignore safeword's own files.

| With Biome        | Without Biome     |
| ----------------- | ----------------- |
| ESLint (devDep)   | ESLint (devDep)   |
| safeword (devDep) | safeword (devDep) |
| jiti (devDep)     | jiti (devDep)     |
| _(no Prettier)_   | Prettier (devDep) |

## SQL Formatting

Your project's chosen SQL formatter owns SQL formatting; safeword adapts to it:

- **If your project formats SQL with Prettier** (`prettier-plugin-sql` in your dependencies), safeword defers entirely: setup skips the SQLFluff configs (`.sqlfluff`, `.safeword/sqlfluff.cfg`), and the edit-time hook runs _your_ Prettier with _your_ config — so `.prettierignore` carve-outs (like frozen DDL migrations) are honored. Adopting `prettier-plugin-sql` in a repo that already has safeword-generated SQLFluff configs? Delete `.sqlfluff` and `.safeword/sqlfluff.cfg`; upgrades won't recreate them.
- **Otherwise**, safeword lints SQL with SQLFluff: the edit-time hook runs `sqlfluff lint` and reports violations to the agent. In-place rewriting is opt-in — set `"sql": { "fix": true }` in `.safeword/config.json` to have the hook run `sqlfluff fix` on edited files. SQLFluff honors `.sqlfluffignore`, so you can exclude files (e.g. applied migrations) from rewriting.

## Hook Configuration

Hooks are configured in `.claude/settings.json` (Claude Code) and `.cursor/hooks.json` (Cursor). Codex uses the profile-scoped Safe Word plugin; install it with `safeword codex install`. See [Hooks & Skills](/reference/hooks-and-skills) for what runs automatically.

To disable a specific hook, remove the safeword hook entry from that agent's config.

### Review gates

Two opt-in review gates (both default off) add independent-review requirements on top of the always-on gates:

- `reviewGate: true` — every ticket phase advance requires an independent phase-exit review stamp (a fresh-context reviewer, logged via `write-review-stamp.ts --phase <phase>`), or an explicitly logged skip reason. Built for autonomous runs where user checkpoints auto-confirm.
- `architectureReviewGate: true` — a new-flow feature can't reach verify/done until its `impl-plan.md` design has been independently reviewed: the Decisions section must cite evidence (URL or `[n]` marker), and the review stamp binds to the plan's content hash — editing the design after review invalidates it. Add `crossModelReview: true` to require the reviewer run on a different model than the author.

### Design approval gate

`designApprovalGate` (default: off) keeps the plan-implementation phase autonomous: the independently reviewed implementation plan advances to TDD without a human checkpoint. Technical builders who want to approve the design before any code can opt in:

```json
{
  "designApprovalGate": true
}
```

When enabled, the agent presents the reviewed plan (riskiest assumption, build order, decisions) and waits for approval before entering the implement phase. In headless/cloud sessions the pending approval is recorded in the ticket work log and the plan surfaces in the session's reviewable output instead of stalling the run.

## Customizing Without Losing Upgrades

Safeword has two types of files:

**Overwritten on upgrade** (you shouldn't edit these directly):

- `.safeword/hooks/*`, `.safeword/guides/*`, `.safeword/templates/*`
- `.claude/skills/*`
- `.cursor/rules/*`, `.cursor/commands/*`

**Never overwritten** (safe to customize):

- `eslint.config.mjs`, `.prettierrc`, `tsconfig.json`, `knip.json`, `cucumber.mjs`
- Language-specific configs: `ruff.toml`, `mypy.ini`, `.golangci.yml`, `clippy.toml`, `rustfmt.toml`, `.sqlfluff`

**Created once, yours afterward** (scaffolded if missing, never overwritten once you've edited them):

- `features/` — Gherkin `.feature` files (a runnable starter is scaffolded)
- `steps/` — TypeScript step definitions for the acceptance lane

**Partially updated** (your customizations preserved, safeword keys re-merged):

- `AGENTS.md`, `CLAUDE.md` — safeword link block re-ensured, your content preserved
- `.claude/settings.json` — safeword hooks updated, your custom hooks preserved
- `.cursor/hooks.json` — safeword hooks updated, your custom hooks preserved
- `package.json` — lint/format/`test:bdd` scripts and devDependencies merged (every project; created minimal + private if missing)
- `.mcp.json`, `.cursor/mcp.json` — MCP server config merged
- `biome.json` — safeword ignore patterns merged (if Biome detected)

**To customize something that gets overwritten:**

1. Copy it to your namespace root (e.g., `.project/my-hook.ts` — see [Project Namespace Root](#project-namespace-root))
2. Update `.claude/settings.json` to point to your copy
3. Your version won't be touched by upgrades

## Project Namespace Root

Safeword's project knowledge — tickets, learnings, personas, glossary, surfaces, architecture record — lives under a single **namespace root** directory, resolved in this order:

1. `paths.projectRoot` in `.safeword/config.json`, if set
2. `.project/` — the default for new installs
3. `.safeword-project/` — the legacy root, honored where it already exists

A project with neither directory resolves to `.project/`.

The `paths` object also supports per-file overrides — `personas`, `glossary`, `surfaces`, and `architecture` — which default to `<root>/personas.md`, `<root>/glossary.md`, `<root>/surfaces.md`, and `<root>/architecture.md` under the resolved root. A per-file key wins over the root for its file.

Persona headers may carry an explicit short code, for example `## Platform Operator (PO)`. Automatic codes use **3–4 letters** (`Platform Operator` → `PLO`); explicit overrides may use **2–4 letters**. Carry the resolved code unchanged through JTBD ids and Gherkin lineage (`oauth-flow.PO1.R1`). Safeword still accepts existing explicit 5–6 character codes and resolves former derived aliases, so upgrading does not require renaming historical specs or scenarios.

Two directory keys, `features` and `steps`, point the BDD acceptance lane at relocated (or host-owned) directories. Unlike the per-file overrides they **augment** the defaults rather than replace them: `safeword codify`, `safeword lint-gherkin`, and the `safeword check` spec gate read the configured `features` directory _in addition to_ root `features/` and the workspace-package conventions, and the scaffolded `cucumber.mjs` runner reads both keys (`steps` matters only to the runner, for relocated TypeScript step definitions). Set them when your acceptance suite lives elsewhere (e.g. `tests/behaviors` + `tests/steps`), including when setup detected an existing cucumber harness and skipped the starter lane:

```json
{
  "installedPacks": ["typescript"],
  "paths": {
    "projectRoot": ".project",
    "personas": "docs/personas.md",
    "glossary": "docs/glossary.md",
    "surfaces": "docs/surfaces.md",
    "architecture": "docs/adr/",
    "features": "tests/behaviors",
    "steps": "tests/steps"
  },
  "docs": {
    "sources": [
      { "type": "local", "path": "README.md" },
      { "type": "local", "path": "docs" },
      { "type": "url", "url": "https://docs.example.com" },
      { "type": "git", "repo": "git@example.com:org/docs.git", "path": "product" }
    ]
  }
}
```

Relative paths resolve against the project root (the directory containing `.safeword/`); absolute paths are used verbatim.

For adopted host harnesses with a house style, `bdd.conventions` points at a conventions document the host owns — what an unimplemented step body looks like, how spec-ahead scenarios are verified without executing them (for example a cucumber dry-run/check profile), and which tags the host's profiles exclude by default:

```json
{
  "bdd": {
    "conventions": "tests/CONVENTIONS.md"
  }
}
```

When set, `safeword codify` prints a pointer to the doc alongside its output, and the installed BDD skill prose directs agents to read and follow it over safeword's defaults. Safeword never parses or validates the doc — it lives next to the harness and is maintained by whoever maintains the harness, so it cannot drift the way conventions transcribed into skill prose do.

`surfaces.md` is optional project knowledge for BDD. Use it for recurring product, agent, runtime, protocol, client, or deployment contexts where behavior must keep working, such as Claude Code, OpenAI Codex, Cursor, web, mobile, MCP, cloud service, or self-hosted deployments. Keep one-off contexts in a feature spec's `## Surfaces` section unless they recur across tickets, are ambiguous enough to drift, or are important enough that missing them would leave a context untested. When a spec lists affected surfaces, saved `.feature` scenarios should cover them with `@surface.<slug>` tags or explicit `skip:` reasons.

`docs.sources` is separate from `paths.*`. Use it to tell audit where project documentation lives. Local sources are validated by `safeword check`; URL and git sources are treated as audit inventory and should be fetched during audit when access is available, or reported as skipped coverage when they are not accessible. Set `"docs": { "sources": [] }` to keep fallback discovery and stop future audits from asking for configured sources.

## Project-Specific Content

Put custom content in your namespace root (`.project/` by default):

- Custom hooks
- Project-specific guides
- Team conventions

This folder is yours—Safeword will never modify it.

## Tracker Integration (`ticketBridge`)

Mirror local tickets one-way into GitHub Issues or Linear. Off by default; configured via
`safeword connect`. See [Tracker Integration](/reference/tracker-integration/) for the full guide.

```json
{
  "ticketBridge": {
    "provider": "none",
    "body": "minimal",
    "target": { "repo": "owner/name" }
  }
}
```

| Key        | Values                       | Meaning                                                                                                                                    |
| ---------- | ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ |
| `provider` | `none` · `github` · `linear` | Which tracker to mirror to. `none` (default) is inert.                                                                                     |
| `body`     | `minimal` · `full`           | Issue body egress. `minimal` (default) = title/status/labels/back-link only; `full` opts in to the ticket body and warns on a public repo. |
| `target`   | `{ repo }` / `{ team }`      | GitHub repo (`owner/name`) or Linear team.                                                                                                 |

Credentials resolve from the OS keychain or an environment variable only — never from this file.
