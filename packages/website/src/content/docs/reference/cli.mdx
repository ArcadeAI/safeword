All commands run via `bunx safeword@latest` (or just `bunx safeword` if you have it installed locally).

## safeword setup

Initialize Safeword in a project. Run this once when you first add Safeword to a repo.

```bash
bunx safeword@latest setup
```

**When to use:** First time setting up Safeword in a project, or after cloning a repo that doesn't have Safeword yet.

### What gets installed

| What                    | Action                                             | Purpose                                            |
| ----------------------- | -------------------------------------------------- | -------------------------------------------------- |
| `.safeword/`            | Creates directory                                  | Hooks, guides, templates                           |
| `.claude/settings.json` | Creates or merges                                  | Hook configuration                                 |
| `.claude/skills/`       | Creates directory                                  | Workflow skills                                    |
| `.agents/skills/`       | Creates directory                                  | Codex workflow skills                              |
| `.codex/config.toml`    | Creates if missing                                 | Codex hook configuration                           |
| `.cursor/hooks.json`    | Creates or merges                                  | Hook configuration                                 |
| `.cursor/rules/`        | Creates directory                                  | Behavior rules                                     |
| `.cursor/commands/`     | Creates directory                                  | Slash commands                                     |
| `eslint.config.mjs`     | Creates file                                       | Linting config (JS/TS)                             |
| `package.json`          | Creates if missing; adds devDependencies + scripts | Quality tools + BDD lane (all projects)            |
| `cucumber.mjs`          | Creates file                                       | Gherkin acceptance-runner config (cucumber-js)     |
| `features/`             | Creates starter                                    | Gherkin `.feature` files — yours after creation    |
| `steps/`                | Creates starter                                    | TypeScript step definitions — yours after creation |
| `.golangci.yml`         | Creates file                                       | Linting config (Go)                                |
| `ruff.toml`             | Creates file                                       | Linting config (Python)                            |
| `mypy.ini`              | Creates file                                       | Type checking (Python)                             |
| `clippy.toml`           | Creates file                                       | Linting config (Rust)                              |
| `rustfmt.toml`          | Creates file                                       | Formatting config (Rust)                           |
| `Cargo.toml`            | Adds lints section                                 | clippy lints (Rust)                                |
| `.sqlfluff`             | Creates file (unless Prettier formats your SQL)    | Linting config (SQL/dbt)                           |
| `AGENTS.md`             | Creates or prepends                                | Links to SAFEWORD.md                               |
| `.mcp.json`             | Creates or merges                                  | MCP server config                                  |
| `.cursor/mcp.json`      | Creates or merges                                  | MCP server config (Cursor)                         |

Setup auto-detects your stack. A TypeScript project won't get Go (beta), Python (beta), Rust (beta), or SQL/dbt (beta) tooling.

The **BDD acceptance lane is universal**: every project gets cucumber-js wired (`test:bdd` script, `features/` + `steps/` starters, `@cucumber/cucumber` + `tsx` + `@types/node` as devDependencies). Step definitions are always TypeScript — they test your app from the outside (shell out to `go test`, `cargo test`, or hit it over HTTP), so a pure Go/Python/Rust repo gets a minimal private `package.json` created to host the lane.

**Unless you already have cucumber**: when setup detects an existing cucumber harness — any `cucumber.{json,yaml,yml,js,cjs,mjs}` it didn't scaffold, or a `@cucumber/cucumber` dependency it didn't add — it skips the starter lane entirely (no files, no deps, no `test:bdd`) and prints the `paths.features` / `paths.steps` config lines that point safeword's BDD tooling at your suite instead. `safeword check` keeps warning until those paths are set, and warns about duplicate lanes left by older safeword versions (it never deletes anything itself). Uninstall/reset likewise leaves your cucumber config untouched; when safeword's own scaffolded lane is present, `reset --full` treats the cucumber devDependencies as safeword's and removes them along with it.

## safeword upgrade

Update Safeword files to the latest version.

```bash
bunx safeword@latest upgrade
```

**When to use:** After a new Safeword release, or when you want the latest improvements. Your customizations to skills and hooks are preserved—only the core files get updated.

## safeword check

Verify your installation and report issues. This verification runs automatically after `setup` and `upgrade`. `setup` and standalone `check` fail closed on broken configuration; `upgrade` still prints post-upgrade diagnostics, but its exit status reflects whether the upgrade apply succeeded.

```bash
bunx safeword@latest check
```

| Option      | Description               |
| ----------- | ------------------------- |
| `--offline` | Skip remote version check |

**When to use:** Standalone use is for CI pipelines and debugging — a health gate in CI, or a diagnostic when something seems broken between upgrades. Shows missing files, version info, and potential problems.

## safeword reset

Remove all Safeword files from your project.

```bash
bunx safeword reset
```

| Option        | Description                           |
| ------------- | ------------------------------------- |
| `--yes`, `-y` | Skip confirmation                     |
| `--full`      | Also remove linting config + packages |

**When to use:** If you want to uninstall Safeword completely. The `--full` flag also removes the linting setup (ESLint config, packages in package.json).

## safeword diff

Show differences between your local files and the latest templates.

```bash
bunx safeword@latest diff
```

| Option | Description           |
| ------ | --------------------- |
| `-v`   | Show full diff output |

**When to use:** Before upgrading, to see what would change. Or to check if you've customized any files.

## safeword sync-config

Regenerate the dependency-cruiser configuration for `/audit`.

```bash
bunx safeword@latest sync-config
```

**When to use:** Before running `/audit`, or after changing your project's architecture boundaries. Detects workspaces and layers, then generates `.dependency-cruiser.cjs`.

## File Ownership

Safeword manages two types of files:

**Owned files** — Safeword controls these completely. They get overwritten on upgrade. Examples: hooks, skills, guides, Cursor commands.

**Managed files** — Created if missing, but never overwritten. You can customize these freely. Examples: `eslint.config.mjs`, `.prettierrc`, `tsconfig.json`, `knip.json`, `.codex/config.toml`, `ruff.toml`, `.golangci.yml`, `clippy.toml`, `.sqlfluff`.

Run `safeword diff` to see which files you've modified.
